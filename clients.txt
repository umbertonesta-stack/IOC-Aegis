DOCUMENTAZIONE CLIENT API - IOC-Aegis
======================================

Questo file documenta la logica interna delle chiamate HTTP dei tre client
attualmente funzionanti in src/ioc_aegis/clients/. Nota: al momento nessuno dei
tre passa dalla cache locale (src/ioc_aegis/cache.py) prima di chiamare l'API:
ogni scansione effettua sempre una richiesta di rete.


1) AbuseIPDBClient (clients/abuseipdb.py)
------------------------------------------
Scopo: verificare la reputazione di un indirizzo IP.

Init:
- legge ABUSEIPDB_API_KEY da variabile d'ambiente (.env); se assente solleva
  ValueError subito, prima di qualsiasi chiamata.
- endpoint fisso: https://api.abuseipdb.com/api/v2/check
- header fissi: Accept: application/json, Key: <api_key>

Metodo check_ip(ip_address) -> IpIOC | None:
1. Costruisce i query params: ipAddress=<ip_address>, maxAgeInDays=90 (considera
   solo segnalazioni recenti, ultimi 90 giorni).
2. GET verso l'endpoint con headers, params, timeout=10s.
3. response.raise_for_status(): se l'HTTP status e' un errore (4xx/5xx), lancia
   eccezione e si passa al blocco except sotto.
4. Estrae data = json["data"] (dict vuoto se assente).
5. Se data e' vuoto: stampa avviso "Nessun dato restituito" e ritorna None
   (non e' un errore, ma l'API non ha info sull'IP).
6. Altrimenti costruisce IpIOC con:
   - value = data["ipAddress"]
   - source = "AbuseIPDB"
   - abuse_score = data["abuseConfidenceScore"]  (0-100, usato direttamente
     come severity score, nessuna trasformazione)
7. Ritorna l'oggetto IpIOC.

Gestione errori:
- requests.exceptions.RequestException (timeout, connessione, HTTP error da
  raise_for_status): stampa messaggio e ritorna None.
- KeyError (l'API cambia formato JSON e manca un campo atteso): stampa
  messaggio e ritorna None.
- ValueError (IpIOC rifiuta uno score fuori range 0-100, difesa a valle):
  stampa messaggio e ritorna None.

In tutti i casi di fallimento il metodo non solleva eccezioni verso il
chiamante: la CLI riceve sempre IpIOC oppure None.


2) URLhausClient (clients/urlhaus.py)
---------------------------------------
Scopo: verificare se un URL e' segnalato come malevolo nel database abuse.ch.

Init:
- legge URLHAUS_API_KEY da variabile d'ambiente; se assente solleva ValueError
  subito.
- endpoint fisso: https://urlhaus-api.abuse.ch/v1/url/
- header fisso: Auth-Key: <api_key>

Metodo check_url(url_address) -> UrlIOC | None:
1. Costruisce il payload di form-data: {"url": url_address}.
2. POST verso l'endpoint con headers, data=payload, timeout=10s.
   (URLhaus usa POST + form-encoded, a differenza di AbuseIPDB che usa GET +
   query string.)
3. response.raise_for_status().
4. Legge data = json completo della risposta.
5. Ramo su data["query_status"]:
   a. "no_results" -> l'URL non e' nel database. Stampa messaggio informativo
      ("probabilmente e' sicuro" - ma e' assenza di prove, non prova di
      assenza) e ritorna None, nessun oggetto creato.
   b. "ok" -> l'URL e' noto. Costruisce UrlIOC con:
      - value = data["url"] (fallback: url_address passato in input)
      - source = "URLhaus"
      - query_status = data["query_status"]
      - url_status = data["url_status"] (online/offline/unknown)
      - threat = data["threat"] (es. "malware_download")
      - tags = data["tags"] (lista, default [])
      Lo score viene poi calcolato da UrlIOC.get_severity_score() in base a
      url_status (online=95, offline=65, unknown=50), non qui nel client.
   c. qualsiasi altro valore di query_status -> stampa "Risposta anomala" e
      ritorna None (difesa contro cambi di contratto dell'API).

Gestione errori:
- RequestException: stampa messaggio e ritorna None.
- ValueError (UrlIOC/IOC rifiuta value o source vuoti): stampa messaggio e
  ritorna None.


3) VirusTotalClient (clients/virustotal.py)
----------------------------------------------
Scopo: verificare quanti motori antivirus segnalano un hash di file come
malevolo.

Init:
- legge VIRUSTOTAL_API_KEY da variabile d'ambiente, ma NON solleva eccezione
  se manca (a differenza degli altri due client) - il controllo e' rimandato
  dentro check_hash().
- endpoint base: https://www.virustotal.com/api/v3
- header fisso: x-apikey: <api_key>

Metodo check_hash(file_hash) -> FileIOC | None:
1. Controllo esplicito: se self.api_key e' vuoto/assente, stampa errore
   critico e ritorna None subito (nessuna chiamata di rete).
2. Costruisce l'URL: {base_url}/files/{file_hash} (l'hash va nel path, non in
   query string o body - MD5/SHA-1/SHA-256 sono tutti accettati da VT come
   identificatore).
3. GET verso l'endpoint con headers, timeout=10s.
4. Caso speciale status_code == 404: hash mai visto da VirusTotal. Stampa
   messaggio informativo e ritorna None PRIMA di chiamare raise_for_status
   (altrimenti il 404 verrebbe trattato come errore generico).
5. response.raise_for_status() per tutti gli altri status non-2xx.
6. Estrae dai dati grezzi:
   - attributes = data["data"]["attributes"]
   - stats = attributes["last_analysis_stats"] (dict per motore: malicious,
     harmless, suspicious, undetected, timeout, ...)
   - voti_maligni = stats["malicious"]
   - voti_totali = somma di TUTTI i valori in stats (non solo malicious+
     harmless: include anche suspicious/undetected/timeout come "motori che
     hanno espresso un verdetto")
7. Costruisce FileIOC con:
   - value = file_hash
   - source = "VirusTotal"
   - malicious_votes = voti_maligni
   - total_votes = voti_totali
   Lo score (FileIOC.get_severity_score()) e' poi malicious_votes/total_votes
   *100, calcolato nel modello, non nel client.

Gestione errori:
- RequestException: stampa messaggio e ritorna None.
- ValueError (FileIOC rifiuta voti negativi o malicious > total): stampa
  messaggio e ritorna None.
  NOTA: nel file attuale il blocco except ValueError e' scritto due volte di
  seguito (duplicato, refuso da correggere) - il secondo blocco e' codice
  morto, non viene mai eseguito perche' il primo except ValueError intercetta
  gia' l'eccezione.


Pattern comune ai tre client
-------------------------------
- Ogni client e' responsabile solo del "come prendere il dato" (HTTP, auth,
  parsing JSON); la logica di "cosa significa il dato" (severity score) resta
  nel modello IOC corrispondente (IpIOC/UrlIOC/FileIOC) - separazione netta
  tra recupero dati e interpretazione.
- Tutti i metodi ritornano l'oggetto IOC concreto oppure None, mai
  un'eccezione non gestita: la CLI non deve mai fare try/except sulle
  chiamate ai client.
- Timeout fisso a 10 secondi su ogni richiesta HTTP.
- Le chiavi API sono sempre lette da variabili d'ambiente (mai hardcoded),
  caricate da .env tramite python-dotenv in cli.py prima di istanziare i
  client.
